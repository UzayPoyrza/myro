# Myro TUI — UX Review (2026-03-09)

35 findings across 10 areas, reviewed against the myro UX skill principles (flow channel, TikTok flow, feedback tiers, scaffolding, navigation, visual design, discoverability, motivation architecture, anti-patterns).

---

## Critical (5)

### 1.1 Post-AC Transition Not Automatic in Chill Mode

**Location:** `app.rs:2506-2515`

On AC verdict in Chill mode, a `Recommend` request is dispatched but the user stays in the editor with their solved code visible. The status message `"✓ accepted!"` auto-clears in ~5 seconds. If the next problem recommendation is still loading when it clears, the screen looks identical to before. The user is never automatically transitioned to the next problem.

In Intense mode, AC triggers a skill deltas popup that requires Enter to continue — a small ceremony but acceptable.

**Violates:** TikTok flow principle — zero-friction automatic transition after AC.

**Recommendation:** In Chill mode, when `ProblemFetched` arrives and the previous problem was AC'd, automatically transition to the next problem. Show a brief 2-3s celebration overlay before the transition.

---

### 2.1 Timer Expiry Blocks ALL Input

**Location:** `app.rs:644-681`, `app.rs:1577-1578`

When the Intense timer expires, `timer_expired = true` and the editor is frozen. All key input is blocked unconditionally — no `/isuck`, no `/quit`, nothing. The status message `"time's up!"` appears for 5 seconds. The user must wait for `RecordAndRefit` to complete, which triggers the skill deltas popup. If `RecordAndRefit` fails silently or takes long, the user is permanently stuck with no escape.

**Violates:** Anti-pattern "block UI during network calls". Fail states must always provide an exit.

**Recommendation:** Continue to accept `/submit` and `/isuck` after timer expiry. Show a clear overlay: `"time expired — /submit to try anyway, /isuck to move on"`. Never block all input unconditionally.

---

### 5.3 UTF-8 Truncation Panic in Past List

**Location:** `ui.rs:1728-1732`

```rust
let title = if entry.title.len() > 20 {
    format!("{:.20}..", entry.title)
} else {
    format!("{:<22}", entry.title)
};
```

`{:.20}` truncates by byte count, not character count. CF problem titles with Cyrillic (e.g., `"Вспышка"`), CJK, or other multi-byte characters will panic if the 20-byte boundary falls inside a multi-byte character. This is the exact gotcha called out in CLAUDE.md.

**Recommendation:** Replace with char-boundary-safe truncation:
```rust
fn truncate_to_chars(s: &str, max_chars: usize) -> String {
    let mut chars = s.chars();
    let mut result: String = chars.by_ref().take(max_chars).collect();
    if chars.next().is_some() { result.push_str(".."); }
    result
}
```

### 6.1 /help is a No-Op

**Location:** `app.rs:1897-1899`

```rust
"help" | "h" => {
    // TODO: show help inline
}
```

The most discoverable command does literally nothing. No feedback, no error message. A user who types `/help` expecting documentation sees the command input disappear with no response — leading them to believe the command system is broken.

**Recommendation:** Implement a scrollable overlay listing all available commands with brief descriptions. Even a static `set_status("commands: /test /submit /coach /hint /skip /quit /debug")` would be better than silence.

---

### 7.3 No Emergency Exit from Intense Mode

**Location:** `app.rs:1870-1872`

Once in Intense mode, `/quit` is blocked. The only options are:
1. Submit and get AC
2. Use `/isuck`
3. Wait for timer to expire (up to 1 hour)

No escape for network issues, wrong problem fetched, or real emergencies.

**Recommendation:** Allow `/quit` in Intense mode with consequences: records as `"abandoned"` with a brief confirmation popup. Gives an emergency exit while preserving mode integrity.

## Moderate (18)

### 1.2 No Difficulty Target Feedback on Home Screen

**Location:** `recommend.rs:296`

The recommender picks problems within `±10%` of `target_probability`. There is no indication on the Home screen or Feed Me menu what target is active. Users see `"P(solve): 70% • 1400"` mid-puzzle but have no context before starting.

**Recommendation:** Show target in Feed Me menu: `"chill  targeting 70% solve rate"`. When no candidates exist, suggest adjusting the target in settings.

---

### 2.2 All Input Silently Dropped During Test Runs

**Location:** `app.rs:1571-1573`

While tests run, all key input is silently discarded. The test panel shows a spinner, but the editor gives no indication it is locked. A user typing in Insert mode gets no feedback that keystrokes are being dropped.

**Recommendation:** Show a dim overlay or change cursor to indicate lock. Allow statement scrolling during test runs. Add cancel capability (Esc or Ctrl+X).

---

### 2.3 AC Celebration Ephemeral and Asymmetric

**Location:** `app.rs:2496`, `ui.rs:1340-1354`

Local test all-pass produces animated sparkle characters. But CF submission AC is only `"✓ accepted!"` in the status line, auto-clearing in 5 seconds. For the culminating moment of a solve session, this is underwhelming. Intense mode gets a skill deltas popup; Chill gets nothing.

**Recommendation:** On Chill AC, show a transient celebration overlay for 2-3 seconds before auto-transitioning. At minimum, keep `"✓ accepted!"` until the next problem loads.

---

### 2.4 Non-AC Verdict Not Forwarded to Coach

**Location:** `app.rs:2530-2532`

On WA, TLE, etc., only `"✗ wrong answer"` is shown. The coach is not notified of the failed CF verdict (only local test failures trigger the coach via `on_test_failure`). A wrong-answer from CF carries valuable coaching context.

**Recommendation:** On non-AC CF verdict, send a `CoachRequest::Analyze` with trigger `"cf_verdict: wrong_answer"`.

---

### 2.5 Status Message Color Inferred from String Prefix

**Location:** `ui.rs:1497-1504`

Status color is determined by whether the string starts with `✓` or `✗`. Messages starting with `"error: ..."` render in `warn_style` (yellow) rather than `fail_style` (red).

**Recommendation:** Add a typed `StatusMessage { text: String, kind: StatusKind }` enum with `Success`, `Failure`, `Warning`, `Info` variants.

---

### 3.2 Auto-Interventions Push-Interrupt Thinking

**Location:** `app.rs:608-626`

The intervention engine fires automatically based on stall/velocity triggers and sends an `Analyze` request to the LLM. This updates the coach panel while the user is in deep focus — push-based, not pull-based as the UX principles require.

**Recommendation:** Either remove automatic LLM interventions (keep engine for state tracking, require `/hint` to pull), or add a config flag `auto_coach = false` that suppresses push interventions.

---

### 3.4 Hint Ladder Progress Invisible

**Location:** `bridge.rs:534`, `bridge.rs:614-622`

The hint ladder advances through `None → Nudge → Question → Formal`. The user has no idea which level they're on or how many hints remain. Once exhausted, `/hint` continues to send LLM requests that return the same terminal message.

**Recommendation:** Show hint progress in coach panel header (e.g., `"coach [hint 1/3]"`). After exhaustion, disable `/hint` and show a status indicating hints are used up.

---

### 4.1 g/G Missing on Most List Screens

**Location:** `app.rs:909-919`

`g`/`G` only work in the debug overlay. Missing from Stats, Past, and ProblemSelect — all of which can have long scrollable lists.

**Recommendation:** Add `g` → scroll to top, `G` → scroll to bottom on all list screens.

---

### 4.3 Tab Key Semantics Conflict

**Location:** `app.rs:1666-1671`, `app.rs:1623-1628`

Tab toggles `statement_focused` when test panel is closed but toggles `output_focused` within the test panel. Two different behaviors on the same key depending on hidden state.

**Recommendation:** Add Tab hint to status line when statement panel is visible. Consider consistent "cycle focus" behavior: editor → statement → test output → editor.

---

### 5.4 Byte-Based Truncation in Debug Log

**Location:** `ui.rs:1426-1429`

```rust
let display = if entry.len() > max_width {
    &entry[..max_width]
} else {
    entry.as_str()
};
```

Indexes by bytes. Will panic on non-ASCII debug messages (e.g., problem titles in log lines).

**Recommendation:** Use char-boundary-safe truncation as in `ghost.rs:46-48`.

---

### 6.3 No "What Next" After Cookie Import

**Location:** `app.rs:1056-1058`

After successful cookie import, app transitions to Home with `"✓ Imported N cookies from Firefox"` that auto-clears. No onboarding guidance for what to do next.

**Recommendation:** After first-time import, highlight "feed me" as the recommended first action or show a brief welcome message.

---

### 7.2 /isuck Framed as "Give Up"

**Location:** `app.rs:2018-2026`

The first-use confirmation popup is titled `"give up"` and describes the command as "marks this problem as too hard." This is shaming language that contradicts the shame-free principle.

**Recommendation:** Retitle to `"skip + learn"` or `"move on"`. Reframe: `"/isuck tells the model this was above your current level — your rating will adjust to suggest better-matched problems next time."`

---

### 7.4 Status Messages Replace All Footer Hints

**Location:** `ui.rs:308-323`, `ui.rs:1793-1797`

When `status_message` is set, it replaces footer navigation hints entirely. User loses all navigation context during errors.

**Recommendation:** Two-line footer: status on top, navigation hints on bottom. Or show status inline without replacing hints.

---

### 8.1 Skill Deltas Only Show in Intense Mode

**Location:** `app.rs:800`

Skill deltas popup gated on `self.last_solve_mode == Some(SolveMode::Intense)`. Chill users never see rating progression after solving — removes a key intrinsic motivator.

**Recommendation:** Show skill deltas in Chill mode too, as a briefer auto-dismissing overlay (2-3 seconds, no blocking).

---

### 9.1 No Elapsed Timer in Chill Mode

No session timer in Chill mode. Chill sessions can go indefinitely with no time awareness. `PastEntry.time_taken_secs` is `None` for Chill sessions.

**Recommendation:** Show subtle elapsed time in status line (e.g., `"23m"`). Track `timer_started` for Chill sessions too.

---

### 9.3 Ctrl+C Quit Loses Work

**Location:** `app.rs:839-847`

Double Ctrl+C sets `should_quit = true` without calling `save_and_finish_past()`. Solution file not saved, past entry stuck as `"in_progress"` forever. `/quit` correctly calls save, but Ctrl+C does not.

**Recommendation:** Call `save_and_finish_past("interrupted")` before setting `should_quit = true`.

---

### 10.1 Recommender Errors Auto-Clear

**Location:** `app.rs:808`

`RecommendEvent::Error` shown via `set_status()` which auto-clears in 5 seconds. Errors like "no candidates in range" require user action but disappear before they can be read.

**Recommendation:** Errors requiring action should persist until dismissed with Enter/Esc.

---

### 10.2 No Retry on Network Fetch Failure

**Location:** `recommend.rs:359-370`

If `fetch_problem_html` fails, the error is shown and cleared. No retry. User must restart the full Feed Me flow from scratch.

**Recommendation:** Add `/retry` command or `r to retry` in footer on fetch failure.

---

## Minor (12)

### 1.3 Feed Me Timer Description Inaccurate

**Location:** `ui.rs:1613-1616`

Feed Me popup describes Intense as `"30-min timer"` but `INTENSE_TIMER_SECS = 3600` (1 hour).

**Fix:** Change to `"60-min timer"` or derive dynamically.

---

### 3.3 Ghost Text Can Overwrite Code

**Location:** `ghost.rs:28`

Ghost text placed on `cursor_row + 1` regardless of whether that line has code.

**Fix:** Only render ghost text when the target line is blank in editor content.

---

### 4.2 Inconsistent List Wrap Behavior

Settings wraps j/k navigation (`(sel + 1) % len`). Stats and Past use `saturating_sub/add` without wrap.

**Fix:** Standardize: small lists (Settings) wrap, long lists (Stats/Past) don't. Document the rule.

---

### 4.4 No Scroll Indicator on Statement Panel

**Location:** `ui.rs:1170-1175`

Paragraph renders with scroll offset but no position indicator. User can't tell how much content is below the fold.

**Fix:** Add `"▼ more"` or `"line N/M"` at bottom-right of statement panel.

---

### 4.5 No Esc-to-Home from Solving Normal Mode

No Esc handler for returning to Home from the Solving screen outside command mode. Users must know `/quit`.

**Fix:** Consider Esc → `/quit` confirmation in Normal mode when no overlay is open.

---

### 5.1 Hardcoded Color in Coach Panel

**Location:** `coach/panel.rs:35`

`ConfidenceLevel::ReadyToHelp` uses `Style::default().fg(Color::Rgb(220, 160, 60))` directly.

**Fix:** Add `ready_style()` to `theme.rs` or reuse `warn_style()`.

---

### 5.2 Hardcoded Editor Selection Background

**Location:** `editor.rs:14`

`.selection_style(Style::default().bg(Color::Rgb(60, 60, 80)))` — not in theme.

**Fix:** Add `SELECTION_BG` to `theme.rs`.

---

### 5.5 Idle Spinner on Home Screen

**Location:** `ui.rs:244, 257`

Spinner always animates on Home header even when nothing is loading. Trains users to ignore spinners.

**Fix:** Only show spinner when a background operation is in progress.

---

### 6.4 /filter and /order Discoverability

Past screen footer shows `/filter  /order` but these require `/` prefix to enter command mode first. Looks like direct shortcuts.

**Fix:** Add direct key shortcuts (`f` for filter, `o` for order) or clarify in footer that these are commands.

---

### 6.5 Empty Past Indistinguishable States

**Location:** `ui.rs:1688-1693`

`"No problems yet. Go solve some!"` shown both for truly empty history and for filters excluding all results.

**Fix:** Distinguish: `"No matches — /filter to adjust"` when filtered out.

---

### 8.2 Stats Has No Progress Narrative

**Location:** `ui.rs:419-530`

Stats shows ratings table and P(solve) bars but no temporal progress. Skill history data exists but isn't displayed.

**Fix:** Add summary at top: `"this week: +12 overall, 3 problems solved"`.

---

### 8.3 Past Re-Open Always Forces Chill

**Location:** `app.rs:1387`

Re-opening from My Past hardcodes `SolveMode::Chill` regardless of original session mode.

**Fix:** Use `entry.mode` to restore original solve mode.

---

### 9.2 No Break Suggestions

No mechanism tracks total session duration or consecutive problems. No break nudges.

**Fix:** After 30 min, subtle footer note. After 10 problems, optional session summary.
