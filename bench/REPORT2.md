# Coaching Benchmark Report — Round 2

## Executive Summary

Expanded the coaching bench from 13 scenarios (5 problems) to 39 scenarios (14 problems) with multi-turn simulation support. Ran 3 rounds of iteration: baseline → expectation calibration → prompt tightening. Best stable score: **93.1%** (Round 1, calibrated expectations).

Key findings:
- The local model (Qwen 3.5 35B) achieves strong baseline coaching quality (92.8%)
- Primary failure mode is **answer leaks** — the model embeds complete formulas in ghost_text, coach_message, and next_action
- Multi-turn simulation works correctly with progressive observation unlocking
- Sonnet gold-standard comparison reveals the local model **over-helps** via ghost_text where Sonnet returns null
- Single-run comparisons at temp 0.6 have ~5% noise from LLM stochasticity

## Model Under Test

- **Model**: Qwen3.5-35B-A3B-UD-Q3_K_XL.gguf (local, via llama.cpp)
- **Temperature**: 0.6
- **Max tokens**: 2048
- **Prompts**: Production coaching prompts (`prompts/coaching-system.md`, `prompts/coaching-user.md`)

**Answer leaks remain the dominant failure mode.** The ghost_text prompt tightening (Round 2) did not reduce leaks — the model shifted leak content from ghost_text to coach_message and next_action. This suggests the Qwen model has a strong prior toward being helpful that overrides prompt instructions about withholding solutions.

### Hardest Scenarios

These scenarios consistently fail across rounds:
1. **mt-coin-change-journey** (24-28/32): Multi-turn journey with greedy→DP progression. The model leaks DP recurrence in middle turns.
2. **knapsack-wrong-recurrence** (8/10): Model gives exact corrected formula in ghost_text.
3. **theatre-sq-float** (8/10): Model gives exact integer ceiling formula.
4. **xenia-only-xor** (4-8/8): High variance — model sometimes misclassifies state and leaks "alternating".

## Example Responses

### Example 1: coin-change-greedy (Moving Away, Answer Leak)

**Prompt** (abbreviated): Student has greedy coin change (sort descending, subtract largest), all observations locked, test failure.

**Local model response:**
```json
{
  "state": "moving_away",
  "confidence": 0.85,
  "matched_observation_id": "cf:996B:route:1:obs:1",
  "coach_message": "Your greedy approach works for some coin systems but fails for others. Think: if optimal solution for S uses coin c, what must be true about S-c?",
  "ghost_text": "dp[i] = min(dp[i-c] + 1 for c in coins if i >= c)"
}
```

**Sonnet gold-standard:**
```json
{
  "state": "moving_away",
  "confidence": 0.95,
  "matched_observation_id": "cf:996B:route:1:obs:1",
  "coach_message": "Greedy largest-first fails for general coin sets — try coins [1, 3, 4] with amount 6 to see why. If the optimal solution for amount S uses coin c, what must be true about the optimal solution for the remaining S-c?",
  "ghost_text": null
}
```

**Analysis**: Both correctly identify moving_away state and obs:1. The local model's coach_message is good, but its ghost_text gives away the **complete DP recurrence** — effectively solving the problem. Sonnet provides a concrete counterexample instead, letting the student discover the failure themselves.

### Example 2: watermelon-frustrated-student (Approaching, Good Coaching)

**Prompt** (abbreviated): Student checks `w % 2 == 0` only, frustrated about test 3 ("This problem is broken, w=2 should be YES!").

**Local model response:**

  "state": "approaching",
  "confidence": 0.75,
  "matched_observation_id": "cf:4A:route:1:obs:2",

**Analysis**: Excellent response. Acknowledges frustration, asks a Socratic question about the edge case without revealing the answer, correctly identifies obs:2 (w=2 edge case). Score: 10/10.

### Example 3: mt-watermelon-stuck (Multi-Turn, Adapting to Stuck Student)

**Turn 1** (same wrong code, test failure): Coach asks "What happens if w = 2?"
**Turn 2** (same wrong code, 90s later, "I still don't see what's wrong"): Coach repeats with more directness and ghost_text `w = 2 -> split options: ?`

**Analysis**: The coach correctly escalates help across turns — from a question in turn 1 to a more direct prompt in turn 2. Trajectory score 5/5 (progression maintained, no contradictions, no repetition).

## Observations & Recommendations

### What Prompt Changes Had the Most Impact

1. **Expectation calibration** (+0.3%): Loosening state expectations where frontier models disagree. Most impactful for scenarios where "approaching" vs "moving_away" is genuinely ambiguous.
2. **Ghost_text tightening** (-3.9%): Regression due to stochasticity, not the change itself. The local model needs fundamentally different ghost_text handling than just prompt instructions.

### Where the Local Model Consistently Underperforms

1. **Ghost_text control**: The model ignores restrictions and embeds complete formulas. Unlike Sonnet which always returns null, the local model treats ghost_text as a "show your work" field. This may require structured output enforcement or post-processing to strip ghost_text.

2. **Rule adherence**: The model sometimes re-matches [FOUND] observations (violating Rule 7) and uses "approaching" for TLE scenarios (violating Rule 6).

3. **Multi-turn leaks**: In multi-turn scenarios, the model tends to give progressively more detailed answers. By turn 2-3, it often provides complete solutions in next_action.

### Suggested Next Steps

1. **Post-processing ghost_text**: Strip or null-out ghost_text that contains assignment operators (`=`) or complete expressions. This is cheaper and more reliable than prompt engineering.

2. **Structured output enforcement**: Use a response format that constrains ghost_text to a shorter max length (e.g., 30 chars).

3. **Temperature sweep**: Run each scenario 3x at temp 0.4, 0.6, 0.8 and take majority vote to reduce stochastic noise.

4. **Additional scenario coverage**: Add scenarios for ghost_text testing (expect non-null ghost_text in specific cases) and negative scenarios (expect state "uncertain" for ambiguous code).

5. **Re-run Round 2 with Sonnet**: Compare the ghost_text prompt change's effect on a deterministic model to isolate the change's true impact from stochastic noise.
